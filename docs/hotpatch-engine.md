# Hotpatch Engine

Status: long-term design target plus shipped conservative implementation.

This document describes the long-term target for Thebe's first-party hotpatch engine. The repository already ships a supported opt-in `thebe dev --hotpatch` path with a local patch server, runtime handshake/control loop, generated `.thebe/hotpatch.rs` glue, text-artifact patch delivery, browser refresh notifications, and restart fallback. This page still captures the fuller target architecture rather than only the currently shipped slice.

## Summary

Thebe now has a conservative server-side hotpatch mode for development that updates a running Axum process without a full restart when a change is safe to patch.

The design will use `subsecond` as the in-process patch runtime, but Thebe itself will own the rest of the system:

1. The development orchestrator.
2. The patch build engine.
3. The patch transport.
4. The patch-vs-restart classifier.
5. The generated runtime glue under `.thebe`.

The reference model is Dioxus `dx`, but the implementation will be narrower and server-only.

## Goals

1. Keep `thebe dev` server-first and compatible with Axum.
2. Make hotpatching a Thebe-owned capability rather than a wrapper around an external CLI.
3. Preserve Thebe's existing codegen boundary: `.trs` changes flow through generated Rust first.
4. Prefer correctness over aggressive patching.
5. Fall back to full restart whenever safety is unclear.

## Non-Goals

1. Browser hot reload for client-side template diffs.
2. WASM, mobile, or desktop runtime support.
3. Multi-process or distributed patching.
4. Cross-workspace patch coordination.
5. Patching release builds.

## User Experience

The intended long-term interface is:

1. `thebe dev` for the normal restart-based loop.
2. `thebe dev --watch` for restart-on-change behavior.
3. `thebe dev --hotpatch` for patch-first development.

Current shipped behavior for `thebe dev --hotpatch`:

1. Run the initial codegen pass.
2. Build and launch the development binary in a hotpatch-aware mode.
3. Watch Thebe inputs and relevant Rust sources.
4. Decide per change whether to patch or restart.
5. Patch route, layout, and component `.trs` template, `<head>`, style, and `<script lang="ts">` changes in place.
6. Restart for Rust source, plain `<script>`, `<script setup>`, shell, config, and dependency changes.
7. Scope layout and component hotpatch delivery to the affected routes when that can be resolved safely.
8. Emit clear diagnostics when a restart is chosen.

The user should not need to install or invoke a second CLI.

## Why Thebe Needs Its Own Engine

`subsecond` is only the runtime side of the system. A usable hotpatch workflow also needs build-side orchestration that can:

1. Produce an initial patchable binary.
2. Capture compiler and linker arguments.
3. Rebuild a thin patch artifact after source changes.
4. Convert that artifact into a jump table.
5. Deliver the patch to the running process.
6. Recognize when patching is unsafe.

That full loop is why the target architecture should follow Dioxus `dx` conceptually instead of depending on a generic external helper.

## Thebe-Specific Constraint

Thebe does not patch `.trs` files directly.

A source edit goes through this pipeline:

1. A `.trs`, `app.html`, `thebe.toml`, or Rust source changes.
2. Thebe regenerates `.thebe/server/**` and any related artifacts.
3. The hotpatch classifier compares the effective Rust delta.
4. If the change is patch-safe, Thebe builds a patch and sends it to the running process.
5. If the change is not patch-safe, Thebe restarts the process.

The true hotpatch boundary is the generated Rust program, not the template source itself.

## High-Level Architecture

The full target system has five parts.

### 1. Dev Orchestrator

The orchestrator is the long-running owner of a hotpatch session.

Responsibilities:

1. Watch the project tree.
2. Run Thebe codegen when relevant inputs change.
3. Track the running child process.
4. Maintain session state for the current executable and patch cache.
5. Call into either the patch engine or the restart path.

The orchestrator should remain in `thebe-cli`, but it should delegate patch logic to dedicated modules.

### 2. Patch Engine

The patch engine owns build-time patch production.

Responsibilities:

1. Resolve the target crate and binary.
2. Perform the initial fat build.
3. Capture enough rustc and linker context to reproduce thin patch builds.
4. Build a patch dylib for eligible changes.
5. Extract a `JumpTable` payload from the patch artifact.
6. Cache per-session build metadata.

This is private infrastructure. It should not become a separate public command surface.

### 3. Runtime Helper

The running development server needs a small hotpatch-aware entrypoint.

Current shipped shape:

1. Generated by Thebe under `.thebe`.
2. Called from the app entrypoint in development mode.
3. Responsible for connecting back to the local patch server.
4. Applies incoming jump tables through `subsecond`.
5. Recreates the top-level server future after a patch.

The app should not hand-maintain patch protocol code.

### 4. Local Patch Transport

The CLI needs a transport between the patch engine and the running process.

Requirements:

1. Local-only in the first milestone.
2. Versioned handshake.
3. Enough metadata to verify the target process matches the active session.
4. Backpressure and failure reporting.

The design does not require HTTP or WebSocket transport. A local TCP protocol is sufficient.

### 5. Patch Classifier

Not every source change is safe to patch.

The classifier decides whether Thebe should:

1. Attempt a patch.
2. Trigger a full restart.
3. Abort and show a hard failure.

The classifier should start conservative and only widen scope after validation.

## File Ownership

Hotpatch support should follow the same ownership model as the rest of Thebe's generated output.

Principles:

1. Generated helper code belongs in `.thebe`.
2. User-authored source files should not accumulate protocol glue by hand.
3. Temporary build metadata should stay out of `src/`.
4. Session caches should be isolated from durable project configuration.

Expected generated/runtime artifacts:

1. `.thebe/server/**` for generated route code.
2. `.thebe/hotpatch.rs` for runtime glue.
3. `.thebe/hotpatch/` for session-local metadata, manifests, and patch cache files.

Exact filenames can be finalized during implementation.

## Development Session Lifecycle

### Startup

When `thebe dev --hotpatch` starts, Thebe should:

1. Load project configuration.
2. Run the normal Thebe codegen pass.
3. Produce an initial patchable development build.
4. Launch the binary with hotpatch runtime support enabled.
5. Start the local patch server.
6. Start file watching.

### Change Handling

For each filesystem event batch, Thebe should:

1. Determine whether Thebe codegen needs to run.
2. Regenerate `.thebe` if needed.
3. Classify the resulting effective Rust change.
4. Choose patch or restart.
5. Surface the reason for that decision.

### Shutdown

On shutdown, Thebe should:

1. Stop the child process.
2. Close the local patch server.
3. Dispose of session-local state.
4. Leave durable generated artifacts in a consistent state.

## Patchable vs Restart-Required Changes

The initial implementation should bias toward restart.

### Likely Patchable

1. Route or component template changes that only affect generated render logic.
2. Layout template changes that only affect routes wrapped by that layout.
3. Route, layout, or component `<head>` changes that only affect generated HTML assembly.
4. Route, layout, or component style changes that only affect generated CSS artifacts.

### Restart Required

1. `Cargo.toml` changes.
2. Dependency graph changes.
3. Any Rust source changes, including non-entry files under `src/`.
4. Any `.trs` `<script>` or `<script setup>` changes.
3. Changes outside the active tip crate, unless explicitly proven safe later.
4. Struct or enum layout changes in live state.
5. Global initialization changes.
6. Signature changes that break the patch boundary.
7. Any patch build failure.
8. Any protocol or session mismatch.

The classifier does not need to be perfect at first. It only needs to avoid unsafe patch attempts.

## Build Model

The build model follows the fat/thin distinction used by existing hotpatch systems.

### Fat Build

The initial build should:

1. Produce the executable that will actually run.
2. Preserve enough symbol and build metadata to support later thin patch builds.
3. Record the compiler, environment, linker, and target context for reuse.

### Thin Patch Build

For a patchable change, Thebe should:

1. Reuse the captured build inputs.
2. Recompile only the required slice.
3. Produce a patch artifact rather than a new full executable.
4. Convert that artifact into a `JumpTable`.

If any of these steps fail or become ambiguous, the engine should force a restart.

## Runtime Integration

The runtime helper should expose a minimal API around the server entrypoint.

Conceptually:

1. The app provides a function that creates the top-level server future.
2. The helper runs that future.
3. The helper listens for incoming patches.
4. When a patch arrives, it applies the jump table.
5. The current top-level future is dropped.
6. A fresh future is created from the same boundary function.

That boundary is important. The system should avoid patching across arbitrary in-flight state.

## Local Protocol

The transport should be explicit and versioned from the start.

### Handshake Metadata

The running process should identify itself with enough information to reject bad patches.

Minimum fields:

1. Protocol version.
2. Session ID.
3. Process ID.
4. Build ID or equivalent binary fingerprint.
5. ASLR reference required by `subsecond`.

### Message Types

The first version only needs a small set of messages:

1. `hello`
2. `patch`
3. `restart-required`
4. `error`
5. `shutdown`

Binary framing is acceptable. Human readability is less important than robustness.

## Configuration

No committed configuration surface is assumed yet.

When configuration is introduced, it should follow these rules:

1. Live in `thebe.toml`.
2. Be optional.
3. Expose only stable user-facing controls.
4. Avoid leaking internal engine details into project config too early.

Reasonable future configuration candidates:

1. Bind address for the local patch server.
2. Restart policy tuning.
3. Verbosity for patch diagnostics.

Configuration should come after the core engine exists.

## Diagnostics And Observability

Thebe should tell the user what it is doing.

Important diagnostics:

1. Initial hotpatch session started.
2. Codegen rerun.
3. Patch attempted.
4. Patch applied successfully.
5. Restart chosen, with reason.
6. Patch failed, with fallback behavior.

The first version does not need a UI beyond clear terminal logs.

## Failure Handling

The system must degrade safely.

Rules:

1. Patch build failure forces restart.
2. Transport failure forces restart or session restart.
3. Runtime patch application failure ends patch mode for that session.
4. Session metadata mismatch rejects the patch.
5. Any uncertain condition should prefer restart.

Hotpatch mode is a productivity feature, not a correctness boundary.

## Platform Scope

The initial target platforms are:

1. macOS.
2. Linux.

Windows should be treated as a later hardening milestone because linker behavior and debug metadata handling will likely need separate work.

## Suggested Internal Module Split

One reasonable layout inside `crates/thebe-cli` is:

1. `hotpatch/orchestrator.rs`
   Session ownership, file watching, restart decisions.
2. `hotpatch/engine.rs`
   Fat build, thin build, artifact handling.
3. `hotpatch/classify.rs`
   Patch-safety rules.
4. `hotpatch/runtime.rs`
   Server-side protocol and patch delivery.
5. `hotpatch/session.rs`
   Session IDs, build fingerprints, cache metadata.

`commands/dev_command.rs` should remain a thin command entrypoint.

## Milestones

### Milestone 1: Minimal First-Party Engine

1. Single local server process.
2. Local transport only.
3. Conservative classifier.
4. Patch a narrow class of handler body changes.
5. Restart for everything else.

Success condition:
A safe route-handler edit updates the running Axum process without a full restart.

### Milestone 2: Better Classification

1. Distinguish template-only generated deltas from state-layout changes.
2. Improve restart reasons.
3. Expand the safe patch set incrementally.

### Milestone 3: Platform Hardening

1. Stabilize macOS and Linux behavior.
2. Add Windows-specific linker and debug handling.
3. Harden cache invalidation and crash recovery.

## Testing Strategy

The implementation should be validated at three levels.

### Unit Tests

1. Classifier rules.
2. Session metadata validation.
3. Protocol framing and parsing.

### Integration Tests

1. Initial session startup.
2. Patch-success path.
3. Restart fallback path.
4. Patch rejection for unsafe edits.

### Manual End-To-End Validation

1. Edit a route handler body.
2. Edit a template-generated render path.
3. Edit a struct layout and confirm restart.
4. Break the patch build and confirm graceful fallback.

## Open Questions

These questions can stay open until implementation starts:

1. Whether the entrypoint bridge should be scaffolded into `src/main.rs` or hidden entirely behind generated code.
2. Whether the local transport should use raw TCP or Unix domain sockets on supported platforms.
3. How much patch cache data should survive across CLI restarts.
4. Whether component/template-only edits can bypass parts of the Rust rebuild pipeline later.

## Decision Record

The current design decisions are:

1. Thebe will own the hotpatch pipeline end to end.
2. `subsecond` remains the runtime patch primitive.
3. Dioxus `dx` is the architectural reference, not a dependency plan.
4. The implementation will start from scratch when work resumes.
5. Safety and restart fallback take priority over patch coverage.
